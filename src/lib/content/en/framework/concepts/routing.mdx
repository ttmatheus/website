# Routing ğŸ›£ï¸

The AzuraJS routing system utilizes a high-performance **Radix Tree** structure. This guarantees constant and ultra-fast lookup times, regardless of the number of registered routes in your application.

## Route Patterns ğŸ§©

Routes are defined within controller methods using decorators to specify paths and parameters.

### Static Routes

Static routes map to fixed paths.

```typescript
@Get("/health")
check() { return "OK"; }
```

### Dynamic Routes

Use the `:` prefix to denote dynamic route parameters.

```typescript
@Get("/users/:id")
getUser(@Param("id") id: string) {
  return { id };
}
```

### Multiple Parameters

Routes can handle multiple dynamic segments.

```typescript
@Get("/org/:orgId/projects/:projectId")
getProject(@Param("orgId") org: string, @Param("projectId") project: string) {
  return { org, project };
}
```

## Supported HTTP Methods ğŸ“¡

AzuraJS provides decorators for all standard HTTP methods:

- `@Get(path)`
- `@Post(path)`
- `@Put(path)`
- `@Delete(path)`
- `@Patch(path)`
- `@Head(path)`
- `@Options(path)`

## Route Priority ğŸ”¢

The routing algorithm applies specific precedence rules to determine which route matches a request:

1.  **Static Routes**: Exact matches like `/users/profile` take precedence over dynamic patterns.
2.  **Parameterized Routes**: Patterns like `/users/:id` are evaluated if no static match is found.
3.  **Wildcards**: Catch-all routes are checked last (if applicable).

## Case Sensitivity ğŸ” 

By default, route matching in AzuraJS is **case-sensitive**. This behavior ensures precise control over URL structures, though it can be adjusted via global configuration (`azura.config.ts`).
